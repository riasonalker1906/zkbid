# Person 1 — Smart Contracts

This folder contains the Hardhat project for the on-chain auction manager.

`AuctionManager` stores bid commitments and encrypted bid blobs during the commit phase. After the deadline, the auctioneer submits the ZK proof generated by the backend. The contract reconstructs the verifier public inputs from on-chain commitments, calls `HonkVerifier.verify`, finalizes the winner/second price, and makes seller payment plus bidder refunds withdrawable.

## File Map

- `contracts/AuctionManager.sol` — main contract. Owns phases, deadlines, bid slots, commitments, encrypted bid storage, verifier call, deposits, refunds, cancellation, and withdrawals.
- `contracts/test/MockVerifier.sol` — simple test verifier used by unit tests so contract logic can be tested without the large generated verifier.
- `test/AuctionManager.test.ts` — Hardhat test suite for bid submission, phase rules, public input construction, finalization, refunds, and cancellation.
- `scripts/deploy.ts` — deploys `AuctionManager` using an already deployed `HonkVerifier` address.
- `hardhat.config.ts` — Solidity compiler, Sepolia network, and Etherscan config.
- `.env.example` — environment variables needed for Sepolia deployment.
- `PERSON1_INSTRUCTIONS.md` — longer implementation and integration notes.

Generated/local files:

- `node_modules/` — npm dependencies.
- `artifacts/`, `cache/`, `typechain-types/` — Hardhat generated outputs.
- `deployments/<network>.json` — deployment output from `scripts/deploy.ts`.

## Contract Interface Used By Teammates

Person 4 backend requires this exact read function:

```solidity
function getCommitments() external view returns (bytes32[] memory);
```

It must always return exactly 5 commitments. Empty slots return:

```text
0x0b63a53787021a4a962a452c2921b3663aff1ffd8d5510540f8e659e782956f1
```

That value is `Poseidon2(0, 0)`, used by the circuit/backend for empty slots.

Main frontend/backend calls:

```solidity
function submitBid(
    uint256 slot,
    bytes32 commitment,
    bytes calldata ephemeralPubkey,
    bytes calldata nonce,
    bytes calldata ciphertext
) external payable;

function finalize(
    uint256 winnerIndex,
    uint256 secondPrice,
    bytes calldata proof
) external;

function withdraw() external;
```

## Local Integration Flow

1. Deploy `HonkVerifier` from `circuit/contracts/ZKVerifier.sol`.
2. Deploy `AuctionManager` with the verifier address.
3. Submit bids to `AuctionManager.submitBid`.
4. Configure Person 4 backend:

```text
COMMITMENTS_SOURCE=ethers
RPC_URL=<local or Sepolia RPC>
AUCTION_ADDRESS=<AuctionManager address>
```

5. Backend calls `getCommitments()`, generates `{ proof, winnerIndex, secondPrice }`.
6. Auctioneer calls `AuctionManager.finalize(winnerIndex, secondPrice, proof)`.
7. Seller and bidders call `withdraw()`.

## Sepolia Deploy

First deploy the generated `HonkVerifier`. This folder does not modify or regenerate `circuit/contracts/ZKVerifier.sol`.

Create `contracts/.env`:

```text
SEPOLIA_RPC_URL=
PRIVATE_KEY=
ETHERSCAN_API_KEY=
VERIFIER_ADDRESS=
AUCTIONEER_ADDRESS=
MIN_DEPOSIT_WEI=0
COMMIT_DURATION_SECONDS=86400
```

Deploy:

```bash
cd contracts
npm run deploy:sepolia
```

Output is written to:

```text
contracts/deployments/sepolia.json
```

## Important Invariants

- Public inputs are built by the contract as `[commitment0..4, winnerIndex, secondPrice]`.
- Caller-supplied public inputs are not trusted.
- Empty slots use the Poseidon zero-bid commitment, not `bytes32(0)`.
- `winnerIndex` must point to a filled slot.
- `secondPrice` must fit in `uint64` and be covered by the winner's deposit.
- Refunds use pull payments through `withdraw()`.
